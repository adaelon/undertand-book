# ADR-0002 知识图谱既是构建产物也是检索结构;砍掉向量索引

状态:已接受(2026-06-21,grill 共识)

## 背景
V2 §3.5 把"向量索引"列为构建产物,是 RAG 条件反射。但 [[ADR-0001]] 锁定本地 plugin 形态后暴露一个现实:Anthropic 无一方 embedding 接口,生成 embedding 必须额外接第三方(Voyage/OpenAI,要 key + 钱 + 书文外传)或在用户机器跑本地模型(下模型 + 占算力)。这是 plugin 形态下**唯一**会压用户硬件 / 逼用户掏外部依赖的环节。Understand-Anything 正因此故意不碰 embedding(默认 Fuse.js 词法搜索)。

## 决策
1. **语义边由 LLM 两遍抽取**:
   - Pass 1 逐窗口抽局部边 + 实体/断言节点(各锚定 LID)→ 沉淀「全局目录」。
   - Pass 2 逐窗口 + 全局目录 → 抽长程边。
   - 每条边过确定性闸:两端 LID 必须真实存在,悬空即丢。
2. **检索(运行时 D 路径)以知识图谱遍历为主**:查询命中的实体/概念/断言 → 顺锚定边跳到其它 LID;是否再外扩由 Claude 读目录后决定(原则二:运行时锚定推理)。
3. **砍掉向量索引**;纯改写、且无任何共享概念锚点的转述,用 **Fuse.js 词法搜索兜底**。

## 命门
向量"搜索"本身极轻(暴力 cosine、毫秒级、无 GPU);真正的负担是"生成 embedding"。砍的是后者带来的外部依赖,不是检索能力——因为图谱给的是"真有关系",比向量的"相似"更贴本项目红线(引用准确率测 citations 的真支持)。

## 否决
- **保留向量 + embedding API**:多一个第三方 key 依赖 + 成本 + 书文外传,破"像 U-A 开箱即用"。
- **保留向量 + 本地 embedding 模型**:下模型 + 构建期占用户 CPU,门槛太高。
- **纯向量 RAG(无图谱)**:相似 ≠ 有关系,抓不到"措辞不同的反驳",且丢掉 §3.5 的概念图谱价值。

## 何时回头
若实测发现"图谱遍历 + 词法兜底"在转述类查询上召回明显不足,再评估把向量作为**可选增强**(默认关、用户自带 key 才开)。
