# ADR-0005 模块 E 存续,重定义为"读时书 agent + 自建最小运行时"

状态:已接受(2026-06-21,grill 共识)。结清 V2 §5 / SESSION_CHECKPOINT 长期悬挂的"模块 E 欠条"。

## 背景
V2 §5 给 E 留欠条:"论证相对裸调 API 的增量,否则降级为参考客户端"。[[ADR-0003]] 定下读时架构后可判。一度倾向"溶解 E",但被需求方纠正:用户要的是**一个承接对本书任何问题的统一入口,带 memory、skills**。

## 决策
**E 存续,但重定义。**
- **E = 读时"这本书的 agent"**:单一入口,承接用户对本书的任何问题;有 **memory**(阅读旅程、历次问答、兴趣与卡点)+ **skills**(可调用能力);**架在 ②③ 之上**(②③ 是它的工具层);跑**用户自选后端**。
- **旧 framing 溶解为 E 的零件**(非独立模块):人格 → 运行时参数(§8);三子代理 → ②③ 路由(工具);跨书 → 加载功能。
- **E 的运行时自建**:本地查询服务内置一个**最小 agent loop**(工具调用 + 记忆),经 provider 抽象对接后端;后端无 tool-calling 时退化 ReAct prompt 循环。(承 [[ADR-0003]])

## 命门
- **这是 U-A 没有、我们净新自建的部分**:U-A 消费端是静态的(dashboard + Fuse.js,读时不跑 LLM、无带记忆 agent)。E 的心脏 = 这个最小 agent 运行时。从 E 起,U-A 不再是参考。
- E 的"真增量"= 多跳自主编排 + 会话态 + 主动策略;裸调 ②③ 是默认基线,E 是其上的带记忆编排者。

## 否决
- **溶解 E 为纯路由**:抹掉了"统一入口 + memory + skills"这一用户核心诉求。
- **委托后端原生 agent API**:行为按 provider 碎裂,排除纯 completion 的简单后端。
- **E 退回 harness(Claude Code/Codex)跑**:违 [[ADR-0003]] 读时脱钩。

## 影响
V2 §2 交付矩阵:E 从"裸三代理壳"改为"读时书 agent + 自建运行时";§5 整节重写。后续待钻:memory 形态、skills 套件。
