# 导读 · 知识图谱与 LID 的关系

> **给谁看**:第一次看 `show-base.ts graph` 输出、看不懂"图谱节点和那串 `@数字` 是什么关系"的人(包括未来的我 / 接手的 agent)。
> **一句话**:**LID 是书的"地址系统";知识图谱是浮在地址之上的一层"语义网"。图谱里的每个东西本身不含原文,它只用 LID 把自己"钉"回书里的某个位置(=锚定)。**
> **配套**:术语精确定义见 `CONTEXT.md`(LID / 锚定 / occurrences / 语义边);设计依据见 `docs/adr/0008`(LID 切分)、`0010`(锚定基数分裂)、`0011`(边作召回路标)。

---

## 1. 两层:分清谁是谁

```
上层:知识图谱   entity / concept / claim ──┐
                                          │ 用 @LID "锚定"(anchor)
下层:LID 地址树  9.9.1  9.9.2  11.18.4 ... ←┘   每个 LID = 书里一个真实段落
```

- **下层 LID**(`9.9.1` 这种点分数字)= **物化路径**,就是地址。`9.9.1` 读作「第 9 个顶层结构 → 第 9 小节 → 第 1 段」。唯一、同级有序、可跳回原文。**确定性切出来,先于图谱存在,与图谱无关**(`[ADR-0008]`)。
- **上层图谱**:LLM 读完书后在这些地址之上"抽"出来的语义结构。图谱节点**自己不存原文**,只记一个名字 + 一串 LID 指针。

`graph` 输出里 `@` 后面那串数字,就是**锚定**——「这个图谱东西出现在书的哪些地址」。

---

## 2. 三种图谱节点:各是什么 + 怎么挂 LID

| 类型 | 是什么 | 真实例子(本书基座) | 锚定方式 |
| --- | --- | --- | --- |
| **entity** 实体 | 反复出现的**具名事物**(一个模式、一个名词) | `命令模式 @11.18.2,11.18.4,11.18.5,11.18.10,11.19.2` | **多锚点** `occurrences:[LID...]` |
| **concept** 概念 | 比实体更抽象的**思想 / 主题** | `简单性 @9.9.1,9.9.2,9.9.3,9.9.6` | **多锚点** `occurrences:[LID...]` |
| **claim** 断言 | 书里说出的**一句具体主张** | `命令模式是无闭包语言模拟闭包的一种方式 @11.18.4` | **单锚点** `source_lid`(只 1 个 LID) |

**为什么 entity/concept 多锚、claim 单锚**(锚定基数分裂,`[ADR-0010]` 决策5):
- 一个**实体**("命令模式")在全书出现多次 → 锚一串 LID。`occurrences` = 「它在书里出现过的所有地址清单」,直接兑现 `book.concept` 的"出现列表"。
- 一句**断言**只在某一段被说出来一次 → 只有 1 个 `source_lid`,指向「这句话是在哪段说的」。

> **跨窗口合并**:`享元模式 @11.19.5,12.10.2,12.10.3,12.10.5` 跨了两个小节。LLM 在两个不同窗口各自认出"享元模式",merge 时按 `id=entity:{规范名}` 把它们的 LID 并成同一实体的清单(`[ADR-0011]` 决策4 的 occurrences 并集)。这是 S3.5 实测验证过的那条。

## 3. edges(边):连的是**节点**,不是 LID

边表达「图谱节点之间的关系」,**不直接锚 LID**:

```
命令模式是无闭包语言模拟闭包的一种方式  →[about 0.85]  命令模式
└── claim 节点                            │            └── entity 节点
                                         └── 关系类型 about,权重 0.85,有向(→)
```

- `→` = 有向边(directed,source→target 有序);`↔` = 无向边(undirected,两端对称,如 `命令模式 ↔[related_to] 闭包`)。
- 边两端是**节点 id**(脚本已把 id 换成名字显示)。
- 边怎么间接碰到 LID?**通过它连的节点**——节点才锚 LID,边把两组地址在语义上连起来。

**关键设计**(`[ADR-0011]` 决策6,会反复碰到):边在读时只当**"召回路标"**——它的作用是告诉系统「要答这个问题该去捞哪些 LID 的原文」,`about`/`related_to` 这些 `edge.type` **不被当权威事实**喂给 LLM,关系对不对让 LLM 自己看原文现判。⇒ 边判错最坏只多捞 / 漏捞原文,**碰不到引用准确性**(结构红线焊死在"真 LID + 真原文")。

## 4. 一个完整例子串起来

拿 LID `11.18.4` 这一段:

```
LID 11.18.4  ←── 地址,指向原文:"这并不是说在其他语言中你不应该使用函数来实现命令模式…"
   ↑ 锚定
   ├── claim:"命令模式是无闭包语言模拟闭包的一种方式"   (source_lid = 11.18.4)
   └── entity:"命令模式" 的 occurrences 之一            (11.18.4 ∈ occurrences)

边再把它们连起来:
   claim ──[about]──→ entity"命令模式"
   claim ──[about]──→ concept"闭包"
```

读这一段时,系统据此知道:这里有一句断言、提到"命令模式"实体和"闭包"概念,且这句断言在讲它俩的关系。**全部靠 LID 把语义钉回原文位置**——这就是「锚定推理 over LID」的物理基础。
