# LLM互动式阅读系统：总体需求规格说明书与实施蓝图（V1.0）

## 1. 宪法级原则（不可违背）

- **原则一：LID唯一语义地基**。全书所有语义切分、数据引用、接口输入输出必须严格锚定Location ID（LID）全局坐标体系，禁止任何模糊定位。
- **原则二：构建期重计算，运行时轻消费**。语义切分、向量索引、知识图谱构建、上下文预组装必须在构建阶段（模块A）100%完成；运行时仅执行无状态查询、调度与消费，禁止运行时重推理。
- **原则三：人机对称消费**。外部Agent与人类读者必须通过同一套标准化API访问内容，不设内部特供通道；Agent接口即书籍数字形态的本体。

## 2. 系统模块定义与交付矩阵

| 模块 | 名称          | 核心定位          | 关键交付物                                                   | 硬性依赖                                 |
| ---- | ------------- | ----------------- | ------------------------------------------------------------ | ---------------------------------------- |
| A    | 预构建基建层  | 系统唯一地基      | LID坐标映射表、向量索引库、知识图谱、上下文预组装包、OpenAPI Schema | 无                                       |
| B    | 实时理解引擎  | 无状态运行时调度  | 查询路由引擎、低延迟消费器、外置Session管理                  | 模块A验收通过                            |
| C    | 定位问答系统  | 业务逻辑中枢      | 意图路由器、上下文组装器、精确引用引擎、API网关              | 模块A验收 + 《数据接口定义说明》四方签署 |
| D    | 开放Agent接口 | 数字感官系统      | 导航/凝视/对话三层API、Agent接入仪式、边界可见性API          | 模块A验收通过                            |
| E    | 书籍专属Agent | 认知肉身/人格化层 | Personality Template、坐标/语境/推理三代理、跨书协议         | 模块A/B/C/D全部验收                      |

## 3. 模块A：预构建产物标准（构建期唯一重计算）

所有产物必须在运行时前固化，形成只读语义基座。

| 产物               | 规格要求                                                     |
| ------------------ | ------------------------------------------------------------ |
| **LID坐标体系**    | 全局唯一，格式`卷.章.节.段.句`；默认粒度到段落，可下探至句子；支持原文双向精准跳转。 |
| **向量索引库**     | 与LID一一绑定；构建期完成Embedding；支持语义相似度检索。     |
| **知识图谱**       | 实体/关系/事件节点必须锚定LID；覆盖率≥90%；支持毫秒级LID↔概念双向查询。 |
| **上下文预组装包** | 每LID节点生成"语境胶囊"：前置背景（必读前文LID列表）+ 横向关联（同主题LID）+ 概念解释（锚定图谱节点）；平均体积适配LLM上下文窗口，API直接拉取。 |
| **接口Schema**     | OpenAPI 3.0规范；包含所有下游模块的输入输出契约；由架构师、预构建负责人、问答系统负责人、开放接口负责人**共同签署**后冻结。 |

## 4. 标准化API契约（模块B/C/D统一接口）

### 4.1 导航层（前庭觉）

**`GET /manifest`**

- **功能**：返回全书语义拓扑与美学节奏。

- 响应关键字段

  ：

  - `structure`: 树状LID拓扑
  - `reading_paths`: 推荐阅读路径
  - `semantic_weight`: 段落语义权重
  - `depth_level`: 认知深度层级

- **验收**：Agent在≤3次调用内建立全书空间心智模型。

### 4.2 凝视层（视觉）

**`GET /context?lid={lid}&granularity={paragraph|sentence}`**

- **功能**：单点聚焦与变焦。

- 响应结构

  ：

  - `near`: 近景（原文片段）
  - `mid`: 中景（前后语境LID及摘要）
  - `far`: 远景（关联概念节点与图谱锚点）
  - 所有片段内嵌`citation_lid`

**`GET /concept?name={name}`**

- **功能**：概念锚点查询。
- **响应**：概念定义 + 出现过的LID列表 + 关联实体。

### 4.3 对话层（语言思维）

**`POST /query`**

- **请求体**：

```json
{
  "session_id": "可选，外置状态",
  "query": "用户/Agent自然语言问题",
  "anchor_lid": "必填，当前关注位置",
  "scope": "auto|local|cross_chapter|global",
  "include_citations": true
}
```

- **响应体**：

```json
{
  "answer": "自然语言答案",
  "citations": [
    {"lid": "24.3.1", "text": "原文片段", "role": "support|contrast"}
  ],
  "related_concepts": ["概念A", "概念B"],
  "suggested_probing": ["追问1", "追问2"],
  "evidence_chain": ["推理步骤LID序列"]
}
```

- **硬性要求**：答案引用准确率≥95%；所有断言必须有LID脚注；人机返回结构完全一致。

**`POST /synthesize`**

- **功能**：跨LID语义编织与综合。
- **请求**：`anchor_lids[]` + `instruction`
- **响应**：同`/query`结构，必须显式绑定多LID证据。

### 4.4 边界与仪式

**`GET /limits`**

- **响应**：速率配额、可访问深度、可写权限、坐标体系版本。

**Agent接入仪式（四步协议）**

1. **握手**：携带API Key，获取`X-Book-Coordinate-System`响应头。
2. **对焦**：调用`GET /manifest`，对齐LID版本与全书拓扑。
3. **漫游**：通过`GET /context`进行探测性调用，建立坐标肌肉记忆。
4. **产出**：正式调用`POST /query`进行语义消费。

## 5. 模块E：书籍专属Agent产品定义

### 5.1 产品本质

专属Agent不是通用LLM套壳，而是基于本书预构建产物长出的**无状态认知代理人**。其全部记忆、方位感与推理能力必须来自模块A/D的标准化接口，禁止自主臆测。

### 5.2 Agent人格模板（构建期注入）

| 类型       | 认知风格   | 输出约束                                                     |
| ---------- | ---------- | ------------------------------------------------------------ |
| **学术型** | 严格、克制 | 强制LID引用；拒绝过度推断；术语必须锚定原文；证据链完整。    |
| **文学型** | 延展、隐喻 | 允许隐喻与联想，但所有关联必须回指知识图谱节点；保留文本细读精度。 |
| **技术型** | 精确、步骤 | 优先公式/代码/步骤；上下文包侧重前置依赖；输出可执行检查清单。 |

### 5.3 三代理机制

- **坐标代理**：通过`GET /manifest`承担人类读者的方位记忆；向人类解释位置时，使用空间隐喻（如"全书最艰深的论证环节"）。
- **语境代理**：通过`GET /context`消费预组装语境胶囊，承担术语翻译、前置背景补全、跨章关联工作；禁止运行时重新推理。
- **推理代理**：面对多跳问题（如"作者是否自相矛盾"），通过`POST /synthesize`提交多LID关联请求，基于预构建图谱完成推理，再降维输出自然语言。

### 5.4 跨书协议（预留）

书A的Agent可通过标准LID接口向书B的Agent发起查询。本书`Manifest`即该Agent的**数字名片**。

## 6. 不可逆实施路线图

```
Phase 0 智脑奠基（阻塞期）
├── 目标：完成模块A全部产物
├── 动作：LID体系固化 → 图谱与向量索引构建 → 上下文包生成 → Schema撰写
└── 准入：架构师、预构建负责人、问答负责人、开放接口负责人共同签署《数据接口定义说明》，冻结基线

Phase 1 神经铺设（并行期）
├── 目标：完成模块B与模块D
├── 动作：实时引擎无状态调度开发 + 开放Agent接口三层API实现
└── 依赖：Phase 0 验收通过

Phase 2 问答中枢（业务期）
├── 目标：完成模块C
├── 动作：意图路由、上下文组装、精确引用引擎、API网关
└── 依赖：Phase 0 签署件 + Phase 1 接口可用

Phase 3 人格外化（产品期）
├── 目标：完成模块E
├── 动作：注入Personality Template → 实现三代理机制 → 跨书协议预留
└── 依赖：Phase 2 精确引用引擎验收通过
```

**顺序铁律**：若缺失模块A的预构建产物（LID、图谱、上下文包），或跳过Schema签署，后续模块一律不得启动。任何跳跃将导致系统不可交付。

## 7. 验收与组织保障

- **本周阻塞动作**：四方负责人必须完成《数据接口定义说明》签署，锁定LID命名规范、上下文包装载结构、图谱查询语法、`POST /query`请求/响应Schema。
- **基线冻结**：签署后，任何Schema变更必须提请评审委员会，禁止口头约定。
- **质量红线**：模块C引用准确率<95%不得进入Phase 3；模块A图谱覆盖率<90%不得进入Phase 1。
